# Seedy: A Chaotic Random Number Generator from Deterministic Thread Contention

**Author:** Sylvain Saucier  
**Technical contributors:** AI semantic prosthesis (code review assistance)  
**Date:** April 2026  

**Document type:** Technical report – no philosophical claims about “true randomness” beyond the chaotic, non‑reproducible behaviour demonstrated.

## Abstract

We present **Seedy**, a minimal random number generator that uses three concurrent threads, three shared 64‑bit state variables arranged in a ring, and no locks, atomics, or explicit random seeds. Despite fully deterministic code, the output on multicore hardware passes standard statistical test suites (PractRand, TestU01) and is non‑reproducible across runs. The mechanism is **chaotic contention**: memory contention between cores creates nanoscale timing variations; the cyclic topology amplifies these variations into macroscopic unpredictability. A Rust port of the generator revealed a hidden undefined behaviour (shift by 64 bits) that was corrected using AI‑assisted code review; the corrected version retains the chaotic properties. This report describes the architecture, empirical validation, and known limitations.

**Keywords:** chaotic RNG, thread contention, memory contention, unstable resonator, Seedy, Rust, undefined behaviour, statistical testing

## 1. Introduction

Random number generation is typically divided into two classes:

- **Pseudorandom (PRNG):** Deterministic, reproducible given a seed.
- **True random (TRNG):** Based on physical noise (thermal, quantum).

Seedy does not fit neatly into either category. It is **deterministic in its code** but **non‑reproducible in practice** due to chaotic sensitivity to thread interleaving and memory contention. We call it a **chaotic random number generator** – a third class that is useful for simulations, games, and any application where non‑reproducibility is desired but cryptographic proof is not required.

## 2. Architecture

### 2.1 State and Threads

- **Three 64‑bit nodes:** `nodes[0]`, `nodes[1]`, `nodes[2]`. Initialised to zero.
- **Three worker threads:** each runs an infinite loop of `seed_modify_64(source, sink)`.
- **Cyclic wiring:**  
  Thread 0: `source = &nodes[0]`, `sink = &nodes[1]`  
  Thread 1: `source = &nodes[1]`, `sink = &nodes[2]`  
  Thread 2: `source = &nodes[2]`, `sink = &nodes[0]`
- **Main thread:** sleeps for an initialisation period (default 1 ms), then repeatedly sleeps for a sampling interval (default 10 µs), reads `nodes[0] ^ nodes[1] ^ nodes[2]` using volatile reads, and writes the bytes to an output buffer only when the value changes.

### 2.2 The Mixing Function

```c
void seed_modify_64(uint64_t *input, uint64_t *output) {
    uint64_t acc = 1;
    for (int x = 0; x < 64; x++) {
        uint64_t val = *input;
        uint64_t rotated = (val << 7) | (val >> (64 - 7));
        *input = rotated;

        // Left rotation of acc by x bits (see §4 for UB fix)
        acc = (acc << x) | (acc >> (64 - x));
        acc *= PRIMES[(2*x) + (rotated & 1)];

        *output += acc ^ rotated;
    }
    *output ^= acc;
}
```

The `PRIMES` array contains 128 pre‑computed 64‑bit primes. The function is fully deterministic.

### 2.3 Volatile Access

All reads and writes to the shared nodes use `read_volatile` / `write_volatile` (in C) or `std::ptr::read_volatile` / `write_volatile` (in Rust). This prevents the compiler from optimising away the memory accesses, ensuring that contention actually occurs.

## 3. Mechanism of Chaotic Contention

### 3.1 Sources of Variation

On a multicore CPU, the following factors cause non‑deterministic timing:

- **Cache coherence protocol** (MESI/MOESI) – the state of a cache line changes unpredictably when multiple cores read/write it.
- **Memory bus arbitration** – the order of accesses is not fixed.
- **DRAM refresh cycles** – row conflicts add variable latency.
- **Clock skew** between cores (nanosecond differences).
- **OS thread scheduling** – the interleaving of threads is non‑deterministic.

When three threads repeatedly read and write the same cache lines (the three nodes are small and likely share a cache line), the system enters a state of **memory contention**. Each thread’s access timing is perturbed by the others.

### 3.2 Cyclic Amplification

The cyclic topology (node0 → node1 → node2 → node0) creates a closed feedback loop. A small timing difference in one thread’s read/write affects the next thread’s input, which is then rotated, multiplied by a prime, and XORed into the output. After 64 rounds per thread and thousands of loops per millisecond, the system’s state becomes **exponentially sensitive** to initial timing conditions – a hallmark of chaos.

### 3.3 Energy Threshold

On a **single core** (or with all threads pinned to one core), there is no true concurrency; the threads run sequentially. In that case, Seedy’s output fails statistical tests. On **two or more cores**, the memory subsystem saturates, and the output passes. This suggests a threshold: when the “energy” (contention) exceeds the system’s dissipation capacity, the chaotic regime emerges. The exact threshold depends on the CPU, cache size, and memory speed.

## 4. The Rust Port and Undefined Behaviour Fix

### 4.1 Porting

A Rust port was created to leverage memory safety while preserving the chaotic behaviour. The port uses `std::ptr::read_volatile` and `write_volatile`, the same cyclic topology, and configurable timings in nanoseconds.

### 4.2 Detection of Shift‑by‑64 UB

During a code review assisted by a large language model (LLM), the rotation line was flagged:

```rust
acc = (acc << x) | (acc >> (64 - x));
```

When `x == 0`, the right shift becomes `acc >> 64`. In Rust (and C), shifting by the bit width of the type is **undefined behaviour**. The compiler may assume this never happens, leading to incorrect optimisations.

The AI suggested replacing it with the well‑defined `rotate_left` method:

```rust
acc = acc.rotate_left(x as u32);
```

This preserves the exact bitwise result for `x` in 1..63 and yields the correct identity for `x == 0` (no change). After applying the fix, the generator was re‑tested with PractRand and TestU01 and passed with no degradation.

### 4.3 Configurable Timings

The Rust version adds `Seedy64::with_timings(init_ns: u64, interval_ns: u64)` allowing the user to control the initialisation sleep and the sampling interval. The default values (1 ms and 10 µs) match the original C behaviour.

## 5. Empirical Validation

### 5.1 Statistical Test Suites

The original C version and the Rust version were tested with:

- **PractRand** (version 0.95) – up to 1 TiB of output. Result: **no failures**.
- **TestU01** (BigCrush) – Result: **passed all tests**.

These are the same batteries used to certify high‑quality PRNGs and some hardware TRNGs. Note: passing these tests does **not** prove true randomness; it only shows the output is statistically indistinguishable from random.

### 5.2 Non‑reproducibility

From the same initial state (all zeros), multiple runs produce different output sequences. This is a necessary condition for a chaotic RNG and is not satisfied by any PRNG.

### 5.3 Single‑core vs. Multicore

On a single‑core machine (or with thread affinity set to one core), the output **fails** the statistical tests. This confirms that memory contention is the essential energy source.

## 6. Comparison with Other RNGs

| Generator | Type | Reproducible? | Requires hardware? | Passes BigCrush? |
|-----------|------|---------------|--------------------|------------------|
| Mersenne Twister | PRNG | Yes (fixed seed) | No | Yes |
| ChaCha20 | PRNG | Yes (fixed seed) | No | Yes |
| Jitterentropy (software) | Chaotic | No | No (uses CPU jitter) | Yes |
| Intel RDSEED | TRNG | No | Yes (CPU) | Yes |
| **Seedy** | Chaotic | No | No (any multicore CPU) | Yes |

Seedy occupies a niche: **software‑only, non‑reproducible, passes statistical tests**. It is not cryptographically proven, but for games, simulations, and non‑adversarial applications it is sufficient.

## 7. Limitations and Open Questions

- **No entropy estimation:** We have not measured the min‑entropy per sample (e.g., using NIST SP 800‑90B). This is required for cryptographic certification.
- **Portability:** Tested on x86_64 (Intel and AMD). Preliminary tests on ARM show similar behaviour, but systematic study is needed.
- **Adversarial predictability:** An adversary with nanosecond‑precision instrumentation could potentially predict outputs. This has not been analysed.
- **Theoretical model:** There is no formal mathematical model (e.g., Lyapunov exponents) of the contention dynamics. The “energy threshold” remains a qualitative concept.
- **Sampling rate:** The 10 µs interval is empirical; faster sampling may produce correlated outputs.

## 8. Conclusion

Seedy is a minimalist demonstration that deterministic concurrent software, when arranged in a cyclic topology with memory contention, can generate statistically random, non‑reproducible output without any dedicated hardware. The Rust port corrects a hidden undefined behaviour and adds configurable timings. While not a cryptographic‑grade TRNG, Seedy is a useful chaotic RNG for many applications and a concrete example of how chaotic dynamics can be harvested in software.
